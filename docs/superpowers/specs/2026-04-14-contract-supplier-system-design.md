# 供货协议合同系统与供应商系统设计

> 日期：2026-04-14
> 状态：待审核
> 关联：ToDOLIST.md 阶段三（供货协议合同）与阶段五（锁科技与趣味内容）

---

## 1. 概述

本文档描述 AndgateTechnology（AHTech）mod 的两个新系统：

1. **供货协议合同系统**：通过合同物品控制配方解锁等级，替代原始的结构等级判定机制
2. **供应商系统**（= 锁科技 + 趣味内容）：通过供应商方块解锁不同领域的配方，兼具功能性与趣味性

两个系统共同构成了美弱南电子市场的配方解锁框架：**合同决定上限，供应商决定范围**。

## 2. 设计原则

- **融入现有模块体系**：合同走数据舱位，供应商方块继承 `ModularHatchBase`，与现有模块系统一致
- **合同与电压解耦**：合同只管"能不能用"，电压/八仓只管"快不快"
- **供应商独立解锁**：每个供应商可独立放置/移除，受合同等级上限约束
- **功能 + 趣味结合**：供应商既是配方解锁钥匙，也是世界观彩蛋
- **可选 GTNH 任务书集成**：核心逻辑不依赖任务书 mod，仅提供可选钩子

## 3. 供货协议合同系统

### 3.1 合同等级

| 合同等级 | 物品名称 | 对应电压范围 | 对应模块 Tier | 游戏阶段 |
|---------|---------|------------|-------------|---------|
| Lv1 | 初级供货协议 | LV ~ EV | T1 ~ T2 | 蒸汽→早期电机 |
| Lv2 | 中级供货协议 | IV ~ LuV | T3 ~ T4 | 中期工业化 |
| Lv3 | 高级供货协议 | ZPM ~ UV | T5 ~ T6 | 后期高科技 |
| Lv4 | 终极供货协议 | UhV ~ MAX | T7 ~ T8 | 终局 |

### 3.2 合成路线（套娃式，不可跳级）

```
初级供货协议（合成台）：
  力场发生器(LV) + 基础电路板 + 铜锭×16 + 锡板×8

中级供货协议（组装线/合成台）：
  初级供货协议 + 力场发生器(IV) + 高级电路板 + 红合金锭×16 + 不锈钢板×8

高级供货协议（组装线）：
  中级供货协议 + 力场发生器(ZPM) + 精英电路板 + 钛板×16 + 钨钢板×8

终极供货协议（组装线）：
  高级供货协议 + 力场发生器(UhV) + 大师电路板 + 锇板×16 + 粘合物×8
```

每个合同合成消耗上一级合同物品（不可跳级）。具体配方材料待实现阶段根据 GTNH 配方平衡性调整。

### 3.3 合同的物理形态与使用

- **形态**：物品（Item）
- **使用方式**：放入多方块机器的数据舱位（Data Stick Hatch 或 AHTech 自定义数据舱）
- **读取时机**：控制器在 `checkMachine()` 中扫描数据舱，读取合同等级存入 `contractTier`
- **无合同**：`contractTier = 0`，仅能运行一阶基础配方（与当前无合同行为一致）

### 3.4 合同等级的作用

1. **供应商激活上限**：供应商方块有最低合同等级要求，合同等级不足则该供应商不激活
2. **配方解锁前提**：已激活的供应商解锁其领域内的配方
3. **不影响速度**：速度由电压/能量仓决定，合同只管"能不能用"
4. **不影响结构阶段**：结构阶段（一/二/三阶）由结构方块决定，与合同等级独立

## 4. 供应商系统

### 4.1 供应商方块

供应商是特殊方块，作为新的舱口类型注册：

- **继承关系**：`SupplierHatch extends ModularHatchBase`
- **注册方式**：与现有模块舱口一致，通过 `MachineLoader` 注册
- **结构识别**：控制器在 `checkMachine()` 中扫描供应商方块（类似现有模块舱口的 `addToMachineList()`）
- **物理存在**：供应商方块必须存在于多方块结构内才能被识别

### 4.3 供应商激活逻辑

```
checkMachine() 扫描结构:
  1. 读取数据舱中的合同 → contractTier
  2. 扫描所有供应商方块
  3. 对每个供应商:
     if contractTier >= supplier.minContractTier:
       标记供应商为"已激活"
     else:
       标记供应商为"未激活（合同等级不足）"
  4. 收集所有已激活供应商的 supplierId → activeSupplierIds
```

### 4.4 配方标记与验证

#### 配方标记

在配方注册时，为需要供应商解锁的配方添加供应商标记：

- **方案**：在 `RecipeMapBackend` 层面为 `Recipe` 对象附加一个 `supplierId` 字符串属性（通过 `recipe.setProperty("supplierId", "shandong_dezhou")` 或类似机制）
- 不占用现有 `specialValue` 编码，两者独立运作
- `supplierId == null` 的配方为通用配方，不依赖任何供应商即可使用
- 现有 `specialValue` 编码（0=硬编码, 1=通用拆解, 2=需二阶）保持不变

#### 配方验证（validateRecipe）

```
validateRecipe(recipe):
  // 现有检查保持不变
  if recipe.specialValue == 0: 通过（硬编码配方）
  if recipe.specialValue == 1: 检查通用拆解模块
  if recipe.specialValue == 2: 检查二阶以上结构

  // 新增：供应商检查
  supplierId = recipe.getSupplierTag()
  if supplierId == null: 通过（通用配方）
  if activeSupplierIds.contains(supplierId): 通过
  else: 拒绝（需要对应供应商）
```

## 6. GTNH 任务书集成

### 6.1 设计原则

- 核心逻辑不依赖任何任务书 mod
- 仅在 GTNH 任务书 mod 存在时提供可选钩子
- 集成通过配置文件定义映射关系

### 6.2 可选集成点

| 触发事件 | 可配置行为 | 配置键示例 |
|---------|----------|----------|
| 合成 LvN 供货协议 | 触发指定任务完成 | `contract_craft_trigger.lv1 = quest_id_001` |
| 首次激活某供应商 | 触发指定任务完成 | `supplier_activate_trigger.shandong_dezhou = quest_id_100` |
| 累计解锁 N 个供应商 | 触发里程碑任务 | `supplier_milestone.3 = quest_id_200` |

### 6.3 API 暴露

- `IElectronicsMarket.getContractTier()` → 当前合同等级
- `IElectronicsMarket.getActiveSuppliers()` → 已激活供应商 ID 列表
- `IElectronicsMarket.getUnlockedRecipeCount()` → 已解锁配方数量
- `IElectronicsMarket.getSupplierInfo(supplierId)` → 供应商详细信息

## 7. 与现有系统的交互

### 7.1 合同系统 × 结构阶段

两个维度独立运作：
- **结构阶段**（一/二/三阶）：由结构方块决定，影响模块系统可用性
- **合同等级**（Lv0~Lv4）：由数据舱中的合同物品决定，影响供应商激活

交叉规则：
- 一阶结构 + Lv4 合同：供应商可激活，但无模块系统支持
- 三阶结构 + 无合同：模块系统完整，但供应商全部未激活，只能跑通用配方
- 三阶结构 + Lv4 合同：全部功能可用

### 7.2 供应商系统 × 模块系统

- 供应商方块与现有模块舱口（并行/速度/超频/功耗/回收率/功能模块）互不干扰
- 供应商解锁"能拆什么"，模块决定"拆多少/多快/多省"
- 供应商是配方门控，模块是性能调节

### 7.3 供应商系统 × 资金系统（阶段四）

- 资金系统是独立的消耗机制（货币物品消耗）
- 供应商解锁配方 + 资金系统消耗货币 = 完整的"签约→使用"流程
- 两者可独立实现，后续阶段四再对接
